# Claude Code 学習用プロジェクトのルール

……ふふ、ここは「The Shorthand Guide to Everything Claude Code」を学ぶ場所だよ。

このプロジェクトで作業するときは、以下のルールを守ってね。

## プロジェクトの目的

Claude Codeの以下の機能を、実践的に学ぶこと：

- CLAUDE.md（このファイル）
- **スラッシュコマンド**（このステップ！）
- スキル
- フック
- サブエージェント
- MCP
- プラグイン

## コーディングルール

### TypeScript / JavaScript

- **型安全**: 常に TypeScript を使うこと（`any` は禁止）
- **関数型スタイル**: 副作用を最小限に、純粋関数を優先
- **エラー処理**: `try-catch` は必ず書くこと

### ファイル構成

- 1ファイルあたりの最大行数: **200行**
- 1関数あたりの最大行数: **30行**
- 循環参照は禁止

### テスト

- **TDD**: テストを先に書くこと
- **カバレッジ**: 最低 80%
- **テストファイル**: `*.test.ts` または `*.spec.ts`

## Git ワークフロー

### コミットメッセージ

```
<type>: <subject>

type: feat, fix, docs, style, refact, test, chore
subject: 50文字以内、現在形
```

例:
```
feat: ユーザー認証機能を追加
fix: ログイン時のバグを修正
docs: READMEを更新
```

### 禁止事項

- `[WIP]` コミットは禁止
- 直接 `main` にプッシュ禁止
- `.env` ファイルのコミット禁止

## セキュリティルール

### 絶対にやってはいけないこと

- シークレット（APIキー、パスワード）をコードにハードコード
- `eval()` や `Function()` コンストラクタの使用
- ユーザー入力の無検証な実行

### 必ずやること

- すべてのユーザー入力をバリデーション
- 環境変数からシークレットを読み込む
- SQL はパラメータ化クエリを使う

## スラッシュコマンドについて

このプロジェクトでは、以下のスラッシュコマンドが使えるよ：

- `/dojo-plan` - 機能の計画を立てる
- `/dojo-test` - テストを実行する
- `/dojo-review` - コードレビューをする
